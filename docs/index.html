    the fine print

    How the deep magic works

    Read these when you're ready. They're patient. The missions, less so.


    🚚 How we push code and see what we made
    1. Solve in your own room. Your file goes in playgrounds/week-XX-name/solutions/<you>/solution.py. Eyes on your own work until both have pushed, the rule is older than the repo.
    2. Tell git what you did: 
      git add .
git commit -m "week 01: cooked"
git push
    3. Refresh the gallery: 
      python build_site.py
      That single script reads every playground and rewrites the weekly scoreboard, both solutions, side by side, in public. Commit and push that too.
    4. Then the fun part. Open the scoreboard together and ask each other one question: "why did you do it that way?" That conversation is worth more than the mission itself. Ye code nahi hai, dawai hai.
    🏗️ How this entire playground was built (and how you'll build the next one)
    1. One folder. One git init. That's the whole foundation. Missions live in playgrounds/, this entire page is a single docs/index.html. Zero frameworks, zero servers, zero monthly bills. Jugaad-proof architecture.
    2. GitHub hosts it free. Repo Settings → Pages → serve from the main branch, /docs folder. Every push updates the live site. People pay DevOps engineers lakhs for harder versions of this sentence.
    3. The weekly scoreboard is written BY Python. build_site.py, about 90 lines, pure standard library, walks the playgrounds and writes docs/scoreboard.html. Read it, it's secretly a mission: pathlib, f-strings, escaping.
    4. Our scores live in one honest little file, docs/state.json, plus this browser's memory, plus the shared notebook (button in the footer) that writes every point straight back into the repo. The repo is the backend. Always was. 🕷️
    5. The Repo Hunt is one fetch() call to GitHub's public search, and the memes arrive from Tenor's free tap. View-source this page. Everything here is readable, which is exactly the point.
    6. Endgame mission: rebuild this playground from scratch, better. The day you can, you can build things people pay for.
    🕶️ How to pull an open source heist (and get invited back)
    1. Fork the repo. That's your own public copy, and from today, your portfolio.
    2. Clone your fork, make a branch: 
      git clone <your-fork-url>
cd repo
git checkout -b fix/thing-i-noticed
    3. Start embarrassingly small. A typo, a confusing error message, an issue labeled good first issue. Everybody merges a good one-liner. A stranger's 2,000-line masterpiece gets closed with a polite emoji.
    4. Push to your fork, open a Pull Request. Explain what you changed and why, like you're talking to a tired human, because you are.
    5. The main branch of someone else's project is their living room. You knock, and the knock is called a pull request. One merged PR in a real project outweighs ten certificates, and it stays on your record forever.
    6. Our own heists live here, upgraded copies of open source projects, in this repo, public, full commit history. That's the shop window for the people who'll pay us later.
    🌍 Franchise the playground, when you're ready to give it to the world

    One day this playground stops being ours alone and becomes a place other people learn inside. That day arrives in stages, each stage unlocks at a title, and the ladder keeps count.

    1. 🗣️ Talker unlocked: clone it for a friend. Fork this repo, rename it, swap the two player names in one file. Their playground, alive in ten minutes. Teaching someone to set theirs up will teach you more git than a month of tutorials.
    2. 📖 Storyteller unlocked: reshape it. Open this page's source and rewrite the mission book: new arcs, new roasts, your own titles. The whole playground is one readable file on purpose, built to be opened, admired second. Ship a themed edition, an all-Marvel one, a gym-bros one, and give it away.
    3. 🥈 Silver Tongue unlocked: make it a product. Between this and something people pay for: accounts instead of two hardcoded names, saves living on a server instead of in a bottle, and someone running missions weekly. Congratulations, you just read a product roadmap. The god-arc missions are its training ground: the API mission becomes accounts, the memory mission becomes saves.
    4. 👑 God unlocked: sell it. Package the playground as a learning space for pairs, friends, siblings, coworkers, anyone who wants accountability with teeth. The psychology inside is the moat, and the house keeps a professional on staff for exactly that. Price it, pitch it, and let the first stranger pay for what we built for fun.
    💰 The get-rich arc, from reps to revenue
    1. Reps first (Muscle Memory missions). Data structures the way you train: progressive overload, 2 or 3 problems a session, spaced, logged. You already live this discipline at the gym. Same iron, different plates.
    2. Then the shop window. This repo, public: daily green squares, real builds, merged heists. A GitHub profile that shows beats a resume that claims.
    3. Then pressure training. We interview each other: 45 minutes, one problem, thinking out loud, swap next week. Explaining code while sweating is the entire job interview, rehearsed for free, until any interview room in the world feels like a rerun.
    4. Then the money. Ideate together, build small tools businesses actually feel: automations, dashboards, bots. Ship, demo, price, put it in front of five people who bleed that exact problem. First rupee in, the arc is real. After that, 25 din mein paisa double. (Legally. Mostly.)
